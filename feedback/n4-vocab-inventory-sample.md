## Native-teacher review notes

- **Honorific / kenjougo entries** like 拝見, ご覧になる, ご主人, ご存じ are **structural** N4 content (introduced in Genki II Lessons 19-20 and Minna II Lessons 49-50). They merit their own thematic section in `vocabulary_n4.md` rather than being scattered across noun / verb sections.
- **Suru-verbs** like 挨拶 / 案内 / 復習 / 反対 follow a predictable noun→suru-verb pattern; the build's POS tagger should mark these with both `noun` and `suru-verb` flags so the runtime can offer either learning mode.
- **Pre-noun adjectival** (連体詞 — `あんな`, `そんな`, `こんな`) is a small POS class (~5 entries) that the N5 build did not formally separate from demonstrative-determiners. The N4 build should adopt the standard Japanese-grammar POS taxonomy explicitly here.
- **Cross-listings** — words like 力 (chikara, "strength") may appear in multiple sections (body-parts adjacent to 体; abstract-properties adjacent to 強い). Use the same slug-encoding rule (§B.1) the N5 build uses.

## Why this is a SAMPLE, not full corpus

Authoring all ~600 N4 vocab entries inline in this manifest would:
1. Be redundant with the authoritative Tanos CSV (which is structured for direct ingestion).
2. Risk staleness: Tanos updates entries occasionally; a snapshotted manifest goes stale.
3. Bloat this repo with N4 content it shouldn't carry.

Instead, this sample demonstrates:
- The format the N4 build will use.
- The POS tagging conventions.
- The borderline / edge-case categories (honorifics, kenjougo, suru-verbs, cross-listings).

The N4 build agent runs `tools/extract_n4_vocab_from_tanos.py` (recipe in Appendix B.12.2) at build time to fetch the full ~600-entry corpus. This sample is the **specification** of what that fetch should produce; the actual data is not in this repo.
